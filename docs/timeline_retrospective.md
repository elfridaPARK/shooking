타임 라인 회고

아래는 쇼핑몰 연동 프로젝트를 기반으로 작성한 타임라인 회고입니다.
모든 이벤트에 대해 영향/통찰/액션 플랜을 작성할 필요는 없으며,
프로젝트 전반에서 나에게 가장 영향 있었던 사건 중심으로 회고를 정리했습니다.

1단계: 프로젝트 기간 동안 중요 이벤트 작성해보기

10월 3일: GitHub · Netlify 배포

10월 10일: 상품 목록 페이지 UI/기능 작성 완료

10월 14일: JEST 기반 단위 테스트 작성

10월 20일: Agile/Scrum 기반 스프린트 진행

10월 23일: 결제 모듈 1차 구현

10월 29일: RESTful API 및 Recoil 상태관리 적용

11월 5일: MSW + Recoil로 API Mocking 완료

11월 7일: React Router 적용 및 페이지 전환 구조 완성

11월 12일: 프로젝트 회고 및 액션 플랜 도출

11월 14일: Cypress를 활용한 쇼핑몰 기능 테스트

11월 20일: 프로젝트 리뷰 및 문서화

11월 21일: 전체 페이지 연동 및 최종 제출

2단계: 이벤트별 영향력 작성

10월 1일 — 요구사항 분석

영향: 프로젝트의 전체 범위와 핵심 목표를 명확히 정의하는 기초가 됨.

10월 3일 — 배포(Github Pages / Netlify)

영향: 실제 서비스 형태로 결과물을 확인하면서 UI·기능 설계 방향성을 잡을 수 있었음.

10월 14일 — 단위 테스트(JEST)

영향: 주요 기능 안정성을 확보하고 에러를 사전에 차단하는 기반이 마련됨.

10월 23일 — 결제 모듈 1차 구현

영향: 프로젝트의 핵심 기능 중 하나로, 페이지 이동/상태관리/데이터 흐름을 전체적으로 이해하는 계기.

11월 5일 — MSW + Recoil Mocking

영향: 실제 API가 없어도 완성형 흐름을 테스트할 수 있어 개발 속도 증가.

11월 14일 — Cypress 기능 테스트

영향: 실제 사용자 흐름 기준으로 전체 페이지 품질을 검증할 수 있었음.

11월 21일 — 전체 페이지 연동

영향: 모든 개발 단계를 하나로 연결하는 최종 핵심 단계로 프로젝트 완성도를 올림.

3단계: 개인적 인사이트 도출

요구사항 분석(10월 1일)

통찰: 개발을 잘하려면 “코드”보다 “문제 정의”가 더 중요하다는 것을 다시 깨달음.
명확한 요구사항이 있을 때 개발 속도가 훨씬 빨라진다는 걸 체감함.

단위 테스트 작성(10월 14일)

통찰: 테스트 코드는 시간이 오래 걸리지만, 이후 개발 단계에서 발생할 문제를 크게 줄여주며
“안정적인 개발 환경”을 만드는 데 반드시 필요함을 느꼈음.

결제 모듈 구현(10월 23일)

통찰: 단순 버튼 클릭 기능보다 데이터 흐름, 로직 설계, 예외처리가 훨씬 중요하다는 걸 실감.
작은 기능 하나도 사용자 경험에 직접적 영향을 준다는 걸 배움.

MSW Mocking 적용(11월 5일)

통찰: 실제 API가 없어도 충분히 개발을 진행할 수 있다는 가능성과
“테스트 우선 개발”이 얼마나 효율적인지 체험함.

Cypress 자동화 테스트(11월 14일)

통찰: 수동 테스트는 놓치는 것이 많고, 자동화 테스트는 장기적으로 큰 비용 절감 효과가 있다는 것을 실감함.

최종 연동 및 제출(11월 21일)

통찰: 컴포넌트 단위 개발보다 “페이지 전체 흐름”을 이해하는 것이 훨씬 더 중요하며,
구조화·문서화가 프로젝트 완성도에 크게 기여함을 배움.

4단계: 액션 플랜 수립

요구사항 분석을 더 체계적으로 정리하기

액션 플랜: 기능 목록, 예외 케이스, 데이터 흐름을 스프린트마다 문서화하고 팀과 공유하기.

테스트 코드 지속적으로 작성하기

액션 플랜: 기능 개발 → 테스트 작성 → 리팩터링 순서를 철저히 지키는 개발 루틴 만들어 적용하기.

상태관리·API 흐름 설계 능력 고도화

액션 플랜: Recoil/Redux Toolkit, React Query 등을 활용한 더 깊은 상태관리 학습 진행.

UI/UX 기준을 명확하게 정하고 개발하기

액션 플랜: Figma를 기반으로 UI 컴포넌트 규칙을 만들고 개발 중 흔들리지 않도록 문서화하기.

자동화 테스트 정착

액션 플랜: Cypress 기반의 e2e 테스트 시나리오를 기능별로 정리하고, 추후 다른 프로젝트에도 적용하기.

정기적인 프로젝트 회고 습관화

액션 플랜: 스프린트마다 회고를 진행해 개발 과정에서 느낀 문제점과 성과를 기록하기.
