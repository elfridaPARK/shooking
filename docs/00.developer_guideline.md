## 2. 시스템 아키텍처

아래는 서비스의 전체 구조를 간략히 표현한 것입니다.

```

Client (Web/App)
↓
API Gateway
↓
Backend Service ───→ External APIs
↓
Database

````

### 기술 스택

- **Frontend:** React / Next.js  
- **Backend:** Node.js (NestJS)  
- **Database:** PostgreSQL + Prisma ORM  
- **Infra:** AWS ECS, RDS, S3, CloudFront  
- **CI/CD:** GitHub Actions  

### 관련 문서

- `/docs/architecture.md` — 시스템 아키텍처  
- `/docs/database-erd.md` — ERD  
- `/docs/api-spec.md` — API 명세

## 3. 개발 환경 설정

### 3.1 시작하기

```bash
# 1. Repository clone
git clone https://github.com/your-org/your-project.git

# 2. 패키지 설치
npm install

# 3. 로컬 서버 실행
npm run dev
````

### 3.2 환경 변수 설정

`.env.example` 파일을 참고하여 `.env`를 생성합니다.

필수 환경 변수 예시:

```
DATABASE_URL=
JWT_SECRET=
AWS_ACCESS_KEY_ID=
AWS_SECRET_ACCESS_KEY=
SERVICE_URL=
```

## 4. 코드 저장소 구조

아래는 전체 디렉토리 구조 예시입니다.

```
/src
  /modules        # 주요 비즈니스 로직(Domain 단위)
  /common         # 공통 예외, 유틸, 인터셉터
  /infra          # DB, 캐시, 외부 연동
  /config         # 환경 설정
/tests            # 테스트 코드
/docs             # 문서 모음
/scripts          # 배포/운영 관련 스크립트
```

* 모듈은 도메인 별로 분리하여 독립성을 유지합니다.
* 서비스 간 의존성이 필요할 경우 Interface 기반으로 접근합니다.

## 5. 브랜치 전략

### 브랜치 타입

* `main`: 배포용(Production)
* `develop`: 통합 개발 브랜치
* `feature/*`: 기능 단위 개발
* `fix/*`: 버그 수정
* `refactor/*`: 구조 개선
* `chore/*`: 설정 변경 등 서비스 로직과 무관한 수정

### PR 규칙

* 최소 1명 리뷰어 승인 필요
* 모든 테스트 통과 후 병합
* 제목은 아래 규칙을 따릅니다.

---

## 6. 커밋 규칙 (Conventional Commits)

```text
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
refactor: 기능 변경 없는 코드 개선
style: 포맷팅/코드 스타일 변경
test: 테스트 코드 관련
chore: 설정/빌드 업무
```

예시:

```
feat(auth): add refresh token API
fix(user): resolve profile update validation bug
refactor(core): restructure service layer for clarity
```

## 7. 프로젝트 핵심 결정 사항

### 중요한 기술적 결정

* **NestJS 채택 이유:** 구조화된 모듈 시스템과 DI 기반 설계
* **Prisma 사용 이유:** 스키마 기반 DB 관리로 유지보수성 향상
* **GitHub Actions 도입:** 테스트 자동화 및 안정적인 배포 흐름 확보

### 구조적 선택

* 기능 단위 모듈 분리로 독립적 개발 가능
* API 응답 포맷 통일(JSON Format)
* 공통 예외 처리(Exception Filter) 적용
